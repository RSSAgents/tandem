Стояла задача - подключить базу данных к уже существующему приложению. До этого вопросы были захардкожены прямо в коде, никакой авторизации не было, очки нигде не сохранялись. Нужно было это исправить.
Выбрали Supabase - это база данных PostgreSQL с готовым API и авторизацией. Не нужно писать отдельный бэкенд.

Первое что встало - где хранить вопросы? Казалось бы, можно просто оставить в коде. Но тогда правильные ответы видны любому кто откроет DevTools. 
Для учебного тренажёра это не критично, но делать правильно приятнее.

Решение оказалось таким - храним вопросы в базе, но создаём специальный View (виртуальную таблицу) `questions_public` которая автоматически убирает поле `correctAnswer` перед отправкой на фронтенд.
Проверка ответа происходит через серверную функцию которая возвращает только `true` или `false`. 
Правильный ответ никогда не покидает базу данных.

Думала как организовать таблицу вопросов. У каждого виджета своя структура  payload - у console один формат, у fill-blanks другой, у stack третий. 
Можно было делать отдельную таблицу для каждого виджета, но решила хранить все в одной таблице `questions` с полем `payload` типа JSONB. 
Это гибко - структура payload может быть любой, и не нужно менять схему базы при добавлении нового виджета.

В итоге получилось три таблицы:
- `questions` — все вопросы всех виджетов
- `questions_public` — View без правильных ответов, именно отсюда читает фронтенд
- `widget_scores` — оценки пользователей за каждый виджет

Головная боль с правильными ответами.
У console виджета `correctAnswer` - это массив строк, порядок которых нужно угадать. Когда попытались передать массив из JavaScript в PostgreSQL функцию, 
получили ошибку - Supabase передавал массив как JSON строку, а PostgreSQL ждал настоящий массив.
Решила поменять тип параметра функции с `TEXT[]` на `JSONB` - тогда оба значения сравниваются как JSON и всё работает корректно.

У fill-blanks другая история — `correctAnswer` это индекс (0, 1, 2, 3) а не текст ответа. Это сделано намеренно чтобы не зависеть от языка - если завтра добавится китайский перевод, логика проверки не сломается.

Переводы.
В приложении уже была настроена локализация через i18next и готовая утилита `getLocalizedString`. Нужно было только правильно организовать данные в базе.

Для fill-blanks виджета текст вопроса и варианты ответов хранятся  как объекты с полями `ru` и `en` прямо внутри JSONB payload:
```json
{
  "text": { "ru": "Функция возвращает {{blank}}.", "en": "The function returns {{blank}}." },
  "options": { "ru": ["число", "строку"], "en": ["number", "string"] }
}
```

На фронтенде просто вызываешь `getLocalizedString(statement.text, lang)` и получаешь нужный язык. Чисто и просто.

Авторизация

Supabase Auth взял на себя всю тяжёлую работу - хранение паролей, сессии, токены. Нам оставалось только подключить это к React.
Сделан `AuthProvider` который слушает изменения состояния авторизации и держит текущего пользователя в контексте. `ProtectedRoute` компонент проверяет есть ли пользователь - если нет, редиректит на главную.
Логин уже был сделан визуально, просто подключила реальный вызов `signIn` вместо заглушки которая просто редиректила на dashboard.

Очки

Каждый виджет теперь считает очки и сохраняет результат в базу:
- Console - 10 очков за вопрос, максимум 100
- Stack - 10 очков за вопрос, максимум 20  
- FillBlanks - 10 очков за вопрос, максимум 80

При повторном прохождении результат просто перезаписывается. 
Использован `upsert` — он либо создаёт новую запись либо обновляет существующую, не нужно проверять существует ли запись.
